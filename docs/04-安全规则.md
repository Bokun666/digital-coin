# 安全规则

## 一、项目安全边界

DigitalCoin 是加密货币交易辅助、AI 分析、策略生成、风控复盘与因子权重演化系统，不是自动交易系统。

MVP 阶段禁止实现以下功能：

1. 自动下单
2. 自动转账
3. 自动提现
4. 保存钱包私钥
5. 保存助记词
6. 浏览器前端保存 API Secret
7. 接入带提现权限的交易所 API Key
8. 未经确认的实盘交易功能
9. 让 GPT 直接执行交易
10. 让 Codex 或 AI 生成的交易代码直接实盘运行

本项目的基本原则是：

> 先保护资产和密钥安全，再考虑功能完整性和开发速度。

---

## 二、资产安全原则

### 1. 禁止保存敏感资产信息

系统禁止保存：

1. 钱包私钥
2. 钱包助记词
3. 交易所提现密钥
4. 交易所 API Secret 的明文前端副本
5. 用户银行卡信息
6. 身份证信息
7. 任何可以直接转移资产的凭证

### 2. 交易所 API 权限原则

交易所 API 权限必须遵守最小权限原则。

阶段规划：

| 阶段 | API 权限 | 说明 |
|---|---|---|
| Phase 1 | 不接 API | 手动录入版 MVP |
| Phase 2 | 公共行情 API | 不需要密钥 |
| Phase 3 | 只读账户 API | 只能读取资产和订单 |
| Phase 4 | 模拟交易 | 不接真实资金 |
| Phase 5 | 是否开放交易权限另行评估 | 必须经过单独安全评审 |

永远禁止：

1. 开启提现权限
2. 保存提现密钥
3. 前端调用带 Secret 的接口
4. 默认开启自动实盘交易
5. 将交易所 API Key 发送给 GPT、Codex 或其他 AI 工具

### 3. API Key 使用规则

如果后续接入交易所或第三方数据平台 API：

1. API Key 只能放在 `.env.local`
2. API Secret 只能在服务端读取
3. 不能在前端代码中出现 Secret
4. 不能在日志中打印 Secret
5. 不能写入 README、docs 或提交记录
6. 如果怀疑泄露，必须立即删除并重新生成
7. 初期交易所 API 必须只读，不允许交易和提现权限

## 三、Git 安全规则

### 1. `.gitignore` 必须包含

```gitignore
node_modules
.next
.env
.env.local
.env.*.local
```

### 2. 必须提交的文件

以下文件必须提交：

```text
pnpm-lock.yaml
package.json
tsconfig.json
next.config.ts
src/
docs/
```

### 3. 禁止提交的文件

以下文件禁止提交：

```text
.env
.env.local
任何包含 API Key 的文件
任何包含 API Secret 的文件
任何包含 Token 的文件
任何包含 Cookie 的文件
任何包含钱包私钥的文件
任何包含助记词的文件
任何包含交易所认证信息的截图
```

### 4. 提交前检查

每次提交前建议执行：

```bash
git status
```

确认没有误提交敏感文件。

必要时可以检查是否误包含关键字：

```bash
git diff --cached
```

重点搜索：

```text
API_KEY
SECRET
TOKEN
PRIVATE_KEY
MNEMONIC
PASSWORD
COOKIE
```

## 四、依赖安全规则

项目使用 pnpm 作为包管理器。

### 1. 依赖安装原则

安装任何新依赖前必须检查：

1. 包名是否正确
2. 是否存在拼写相似包风险
3. 是否为主流包
4. 是否长期维护
5. 是否有异常低下载量
6. 是否近期出现异常版本发布
7. 是否包含 postinstall/build script
8. 是否涉及钱包、交易所、签名、密钥、自动交易等敏感功能
9. 是否真的有必要安装
10. 是否可以用项目已有能力替代

### 2. 禁止行为

禁止：

1. 复制网上命令直接安装陌生包
2. 无脑批准 build scripts
3. 随便删除 `pnpm-lock.yaml`
4. 一次性安装大量不必要依赖
5. 使用来源不明的交易所 SDK
6. 使用来源不明的钱包 SDK
7. 使用无人维护的安全敏感包
8. 为了省时间绕过依赖安全检查

### 3. 依赖选择优先级

优先使用成熟、主流、维护活跃的依赖。

第一阶段尽量只使用：

1. next
2. react
3. react-dom
4. typescript
5. tailwind
6. prisma
7. @prisma/client
8. zod
9. react-hook-form
10. recharts

能不用第三方包就不用。

仓位计算、风险评分、因子权重计算等核心逻辑，优先自己实现，不依赖冷门包。

## 五、pnpm 安全规则

### 1. 检查被忽略的构建脚本

```bash
pnpm ignored-builds
```

### 2. 审批构建脚本

```bash
pnpm approve-builds
```

注意：

> 只批准明确知道用途的依赖。

例如 Next.js 初始化中出现的 `sharp`、`unrs-resolver` 需要确认来源后再批准。

### 3. 安全扫描

```bash
pnpm audit
```

如果发现漏洞：

1. 先确认漏洞影响范围
2. 查看是否来自直接依赖还是间接依赖
3. 优先升级相关依赖
4. 如果是框架内部依赖且当前业务不触发漏洞条件，可以记录到安全待办
5. 不要盲目降级或随意删除 lockfile

### 4. 严格按 lockfile 安装

```bash
pnpm install --frozen-lockfile
```

生产部署、换电脑安装、CI 环境建议使用该命令。

## 六、环境变量规则

### 1. 环境变量文件

本地开发使用：

```text
.env.local
```

示例：

```env
DATABASE_URL="file:./dev.db"
OPENAI_API_KEY="your_api_key_here"
```

### 2. 前端暴露规则

Next.js 中，只有 `NEXT_PUBLIC_` 开头的变量会暴露到浏览器。

禁止这样写：

```env
NEXT_PUBLIC_EXCHANGE_SECRET="xxx"
NEXT_PUBLIC_OPENAI_API_KEY="xxx"
NEXT_PUBLIC_API_SECRET="xxx"
```

允许前端暴露的只能是无敏感信息的配置，例如：

```env
NEXT_PUBLIC_APP_NAME="DigitalCoin"
```

### 3. 服务端读取规则

API Key、API Secret、数据库连接等敏感信息只能在服务端读取。

包括：

1. Route Handler
2. Server Action
3. 后端 service
4. 脚本任务

禁止在 Client Component 中读取密钥。

## 七、GPT / AI 安全规则

本项目会接入 GPT 作为 AI 分析助手，因此必须明确边界。

### 1. GPT 的角色

GPT 可以作为：

1. 市场信息总结员
2. K 线截图分析员
3. 宏观风险分析员
4. 合约数据分析员
5. 链上数据分析员
6. 策略参数建议员
7. 复盘辅助分析员
8. 用户判断对比分析员

GPT 不能作为：

1. 自动交易执行员
2. 资产托管者
3. 账户控制者
4. 提现执行者
5. 无需用户确认的交易决策者

### 2. 禁止发送给 GPT 的内容

禁止发送：

1. 钱包私钥
2. 助记词
3. 交易所 API Secret
4. 提现密钥
5. 完整认证 Header
6. Cookie
7. 身份证信息
8. 银行卡信息
9. 未打码的敏感截图
10. 任何可以直接控制资产的信息

### 3. GPT 输出必须经过风控

GPT 给出的任何交易建议或策略参数，都必须进入以下流程：

```text
AIAnalysis
  ↓
TradePlan
  ↓
RiskCheck
  ↓
PositionSizing
  ↓
用户手动确认
```

GPT 不能绕过：

1. 交易计划
2. 风险检查
3. 仓位计算
4. 用户确认

### 4. GPT 分析结果标识

系统中所有 GPT 输出必须明确标识为：

```text
AI 分析建议，不构成投资建议，不代表系统要求执行。
```

## 八、交易安全规则

### 1. 交易计划必须先于交易

每次交易前必须先创建 TradePlan。

TradePlan 至少包括：

1. 币种
2. 操作类型
3. 方向
4. 入场理由
5. 入场价
6. 止损价
7. 止盈价
8. 投入金额
9. 最大可接受亏损
10. 杠杆倍数
11. 保证金模式
12. 当前情绪状态

### 2. 合约交易限制

MVP 阶段，合约只作为学习和记录场景，不做自动交易。

合约交易默认规则：

1. 优先逐仓
2. 不建议全仓
3. 不建议高杠杆
4. 没有止损不建议开仓
5. 重大宏观数据前不建议重仓
6. 情绪状态为 FOMO、冲动、想回本时不建议操作
7. 单笔最大可接受亏损必须提前定义

### 3. 风险评分拦截

当 RiskCheck 判断为极高风险时，系统应明确提示：

```text
不建议操作。
该交易可能属于情绪化交易、无止损交易或过度杠杆交易。
```

后续如果实现按钮逻辑，极高风险计划可以要求用户二次确认，但 MVP 阶段不做自动交易。

## 九、日志安全规则

日志中禁止打印：

1. API Key
2. API Secret
3. Token
4. Cookie
5. 私钥
6. 助记词
7. 交易所完整认证 Header
8. 用户敏感身份信息

错误日志应只记录必要信息。

错误示例：

```text
OKX API request failed: 401 Unauthorized
```

不要记录：

```text
OKX API request failed, apiKey=xxx, secret=xxx
```

## 十、数据库安全规则

### 1. 不保存高危敏感信息

数据库禁止保存：

1. 钱包私钥
2. 助记词
3. 交易所 API Secret 明文
4. 提现密钥
5. 身份证号
6. 银行卡号

### 2. 金额字段精度

金额、价格、仓位、盈亏等字段不能使用普通 float。

应使用：

1. Decimal
2. 字符串金额
3. 数据库高精度数值类型

避免因为浮点误差导致计算错误。

### 3. 重要记录保留历史

以下内容必须保留历史：

1. 风险检查结果
2. AI 分析结果
3. 用户判断
4. 策略参数
5. 因子权重变化
6. 交易复盘

不能只覆盖最新值，否则无法复盘。

## 十一、Codex / AI 开发规则

使用 Codex 或其他 AI 辅助开发时，必须遵守：

1. 不向 AI 提供真实 API Key
2. 不向 AI 提供真实 API Secret
3. 不向 AI 提供钱包私钥
4. 不向 AI 提供助记词
5. 让 AI 新增依赖前，必须说明原因
6. AI 生成的依赖安装命令必须人工确认
7. AI 生成的安全相关代码必须人工 review
8. AI 生成的自动交易代码不能直接实盘运行
9. AI 生成的环境变量示例只能使用假值
10. AI 不能绕过项目安全规则

给 Codex 的默认要求：

```text
请严格遵守 docs/04-安全规则.md。
不要引入不必要依赖。
不要实现自动下单。
不要保存私钥、助记词或交易所 Secret。
新增依赖前必须说明原因。
```

## 十二、截图安全规则

如果后续上传截图给系统或 GPT 分析，必须注意：

1. 打码 API Key
2. 打码账户 UID
3. 打码邮箱
4. 打码手机号
5. 打码实名信息
6. 打码完整钱包地址，除非确实需要分析链上地址
7. 打码交易所资产敏感页面中的隐私信息

K线图、公开行情图、公开数据图一般可以直接分析。

账户资产截图、API 页面截图必须谨慎。

## 十三、数据来源安全规则

后续接入外部数据源时，优先级如下：

1. 官方公开 API
2. 知名数据平台公开 API
3. 只读 API
4. 用户手动录入
5. 用户截图输入

谨慎使用：

1. 来历不明的第三方 API
2. 私人维护的免费接口
3. 需要提供交易所密钥的聚合服务
4. 不透明的行情 SDK
5. 未知 npm 包提供的数据服务

## 十四、应急处理规则

### 1. API Key 泄露

如果怀疑 API Key 泄露：

1. 立即删除该 API Key
2. 重新生成新 Key
3. 检查是否开启交易或提现权限
4. 检查交易所账户异常登录
5. 检查是否有异常订单
6. 检查 Git 提交历史
7. 必要时更换相关密码和 2FA

### 2. 误提交敏感文件

如果误提交 `.env.local` 或密钥：

1. 立即删除远程仓库中的敏感内容
2. 立即废弃泄露的 Key
3. 重新生成 Key
4. 清理 Git 历史
5. 强制推送前先确认不会影响其他协作者
6. 不要认为删除文件就安全，提交历史中仍可能存在

### 3. 依赖疑似恶意

如果发现依赖可疑：

1. 立即停止运行项目
2. 删除 node_modules
3. 检查 package.json
4. 检查 pnpm-lock.yaml
5. 回滚到安全提交
6. 重新安装依赖
7. 检查本机是否存在异常环境变量泄露

---

## 十五、安全优先级

本项目优先级如下：

1. 资产安全
2. 密钥安全
3. 依赖安全
4. 数据准确性
5. 风控逻辑
6. 复盘有效性
7. 策略可解释性
8. 功能完整性
9. 页面美观

任何功能都不能牺牲资产安全和密钥安全。

如果某个功能需要在“安全”和“便利”之间选择，默认选择安全。

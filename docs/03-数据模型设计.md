# 数据模型设计

## 一、文档目的

本文档用于描述 DigitalCoin 项目的核心数据模型设计。

DigitalCoin 是一个加密货币交易辅助、AI 分析、策略生成、风控复盘与因子权重演化系统。

本系统第一阶段以手动录入为主，不接交易所交易接口，不自动下单，不保存钱包私钥、助记词或交易所提现权限相关信息。

数据模型设计目标：

1. 支撑自选币、交易计划、风险检查、仓位计算、交易记录、复盘等 MVP 功能。
2. 支撑后续 GPT 分析、用户判断对比、策略生成、参数推荐。
3. 支撑因子权重演化、策略复盘和长期优化。
4. 保证资产安全和密钥安全。
5. 尽量让核心模型清晰、可扩展、便于后续用 Prisma 实现。

---

## 二、设计原则

### 1. 金额和价格使用高精度类型

所有金额、价格、数量、盈亏、比例相关字段，在数据库层建议使用 Decimal。

不要使用普通 float 直接处理金额和币价，避免精度误差。

例如：

- plannedAmount
- entryPrice
- stopLossPrice
- takeProfitPrice
- profitLossAmount
- totalAssetUsdt
- totalAssetCny

---

### 2. 交易计划和交易记录分离

TradePlan 表示“操作前计划”。

TradeRecord 表示“真实执行结果”。

二者必须分离。

原因：

1. 用户可能创建计划但最终不执行。
2. 用户实际执行价格可能与计划价格不同。
3. 计划可以用于风险检查。
4. 真实结果可以用于复盘和策略优化。

---

### 3. 风险检查结果要留痕

RiskCheck 不应该只是 TradePlan 的一个字段。

每次风险检查都应该保存一条记录。

原因：

1. 风险规则可能会变化。
2. 同一个计划可能多次检查。
3. 后续复盘时需要知道当时系统为什么提示高风险。
4. 方便分析哪些风险提示真正有效。

---

### 4. AI 分析结果要保存

GPT 的分析结果不能只显示在页面上，必须保存。

保存内容包括：

1. 输入摘要
2. GPT 总结
3. 偏多/偏空/震荡判断
4. 置信度
5. 风险点
6. 参数建议
7. 原始回复

这样后续才能判断：

1. GPT 当时是否判断正确。
2. 用户判断和 GPT 判断谁更接近真实结果。
3. GPT 在哪些行情下容易误判。
4. GPT 建议的参数是否有实际价值。

---

### 5. 因子权重要有历史

Factor 当前权重只是当前状态。

每次调整权重都必须写入 FactorWeightHistory。

否则无法追踪：

1. 为什么调整权重。
2. 谁确认调整。
3. 调整前是多少。
4. 调整后是多少。
5. 这次调整是否带来效果。

---

### 6. 安全字段禁止入库

系统禁止保存以下内容：

1. 钱包私钥
2. 助记词
3. 交易所 API Secret
4. 提现密钥
5. 完整认证 Header
6. 未打码的敏感身份信息
7. 银行卡信息

如果未来需要接入交易所账户，只能从只读 API 开始，并且敏感信息只能放在服务端环境变量中。

# 1. WatchCoin 自选币

## 作用

用于记录用户关注的币种、关注理由和风险标签。

该实体是机会池的基础。

## 字段设计

| 字段名 | 类型 | 说明 |
|---|---|---|
| id | String | 主键 |
| symbol | String | 币种符号，如 BTC、ETH、KAS |
| name | String | 币种名称 |
| category | CoinCategory | 币种分类 |
| watchReason | String | 关注理由 |
| supportPrice | Decimal | 关键支撑位 |
| resistancePrice | Decimal | 关键压力位 |
| riskTags | String | 风险标签，多个标签可用逗号分隔 |
| status | WatchStatus | 状态 |
| notes | String | 备注 |
| createdAt | DateTime | 创建时间 |
| updatedAt | DateTime | 更新时间 |

## 示例

```text
symbol: KAS
category: MINING
watchReason: 鱼池上线 KAS 矿池，可能带来矿工关注和算力增长
riskTags: 矿工抛压, 山寨币, 高波动
status: WATCHING
```

# 2. TradePlan 交易计划

## 作用

用于记录交易前的完整计划。

TradePlan 是系统最核心的实体之一。

所有交易建议、GPT 参数推荐、策略执行，都应该先落到 TradePlan，再进入风险检查和仓位计算。

## 字段设计

| 字段名 | 类型 | 说明 |
|---|---|---|
| id | String | 主键 |
| coinSymbol | String | 币种 |
| watchCoinId | String | 关联自选币，可为空 |
| operationType | OperationType | 操作类型 |
| direction | TradeDirection | 方向 |
| plannedAmount | Decimal | 计划投入金额 |
| totalCapital | Decimal | 当前总资金 |
| currentPrice | Decimal | 当前价格 |
| entryPrice | Decimal | 计划入场价 |
| stopLossPrice | Decimal | 止损价 |
| takeProfitPrice | Decimal | 止盈价 |
| leverage | Decimal | 杠杆倍数 |
| marginMode | MarginMode | 保证金模式 |
| reason | String | 入场理由 |
| maxAcceptableLoss | Decimal | 最大可接受亏损 |
| hasMacroEvent | Boolean | 是否有重大宏观事件 |
| isChasingPrice | Boolean | 是否追涨 |
| hasStopLoss | Boolean | 是否设置止损 |
| emotionState | EmotionState | 情绪状态 |
| status | TradePlanStatus | 计划状态 |
| createdAt | DateTime | 创建时间 |
| updatedAt | DateTime | 更新时间 |

## 关键规则

1. 合约交易必须填写 leverage。
2. 合约交易必须填写 marginMode。
3. 如果 hasStopLoss 为 false，风险评分必须显著增加。
4. 如果 emotionState 为 FOMO、REVENGE、IMPULSIVE，应视为高风险。
5. 如果 maxAcceptableLoss 超过 totalCapital 的 2%，应提示风险过高。

# 3. RiskCheck 风险检查

## 作用

根据 TradePlan 自动生成风险评分和风险原因。

风险检查结果必须保存历史。

## 字段设计

| 字段名 | 类型 | 说明 |
|---|---|---|
| id | String | 主键 |
| tradePlanId | String | 关联交易计划 |
| score | Int | 风险评分，0-100 |
| level | RiskLevel | 风险等级 |
| reasons | String | 风险原因 |
| suggestion | String | 建议 |
| createdAt | DateTime | 创建时间 |

## 初始评分规则

| 条件 | 加分 |
|---|---:|
| 没有止损 | +25 |
| 合约且杠杆大于 3 倍 | +20 |
| 合约且使用全仓 | +25 |
| 情绪状态为 FOMO / REVENGE / IMPULSIVE | +25 |
| 有重大宏观事件 | +15 |
| 追涨 | +15 |
| 最大亏损超过总资金 2% | +20 |

## 风险等级

| 分数 | 等级 |
|---|---|
| 0 - 30 | LOW |
| 31 - 60 | MEDIUM |
| 61 - 80 | HIGH |
| 81 - 100 | EXTREME |

极高风险时，系统应该提示：

```text
不建议操作。该计划可能存在过度杠杆、无止损、追涨或情绪化交易问题。
```

# 4. PositionSizing 仓位计算

## 作用

用于保存一次交易计划对应的仓位计算结果。

它的核心问题是：

```text
如果这笔交易失败，最多会亏多少钱？
```

## 字段设计

| 字段名 | 类型 | 说明 |
|---|---|---|
| id | String | 主键 |
| tradePlanId | String | 关联交易计划 |
| positionValue | Decimal | 实际仓位价值 |
| lossAmountAtStop | Decimal | 止损亏损金额 |
| profitAmountAtTakeProfit | Decimal | 止盈盈利金额 |
| riskRewardRatio | Decimal | 盈亏比 |
| lossPercentOfTotalCapital | Decimal | 止损亏损占总资金比例 |
| suggestion | String | 仓位建议 |
| createdAt | DateTime | 创建时间 |

## 初始计算逻辑

```text
positionValue = plannedAmount × leverage
lossAmountAtStop = abs(entryPrice - stopLossPrice) / entryPrice × positionValue
profitAmountAtTakeProfit = abs(takeProfitPrice - entryPrice) / entryPrice × positionValue
riskRewardRatio = profitAmountAtTakeProfit / lossAmountAtStop
lossPercentOfTotalCapital = lossAmountAtStop / totalCapital × 100
```

注意：

1. 现货交易 leverage 可以视为 1。
2. 稳定币理财 leverage 可以视为 1。
3. 如果没有止损价，无法计算真实最大亏损，应提示风险。
4. 实际合约强平计算比该公式复杂，后续可以单独扩展 LiquidationRisk 模块。

## 作用

用于记录真实交易结果。

它和 TradePlan 分离。

## 字段设计

| 字段名 | 类型 | 说明 |
|---|---|---|
| id | String | 主键 |
| tradePlanId | String | 关联交易计划，可为空 |
| coinSymbol | String | 币种 |
| operationType | OperationType | 操作类型 |
| direction | TradeDirection | 方向 |
| entryTime | DateTime | 入场时间 |
| exitTime | DateTime | 出场时间 |
| entryPrice | Decimal | 入场价 |
| exitPrice | Decimal | 出场价 |

| leverage | Decimal | 杠杆倍数 |
| profitLossAmount | Decimal | 盈亏金额 |
| profitLossPercent | Decimal | 盈亏比例 |
| followedPlan | Boolean | 是否遵守计划 |
| exitReason | String | 出场原因 |
| createdAt | DateTime | 创建时间 |
| updatedAt | DateTime | 更新时间 |

## 关键规则

1. TradeRecord 可以不关联 TradePlan，但不推荐。
2. 如果 followedPlan 为 false，该交易不应该直接用于评估策略因子有效性。
3. 交易记录是复盘和统计的基础。

# 6. Review 交易复盘

## 作用

用于记录交易后的复盘。

Review 是系统长期价值的核心数据来源。

## 字段设计

| 字段名 | 类型 | 说明 |
|---|---|---|
| id | String | 主键 |
| tradeRecordId | String | 关联交易记录 |
| followedPlanReview | String | 是否遵守计划的复盘 |
| emotionReview | String | 情绪状态复盘 |
| mistake | String | 错误总结 |
| lesson | String | 经验教训 |
| nextAction | String | 下次改进 |
| createdAt | DateTime | 创建时间 |
| updatedAt | DateTime | 更新时间 |

## 复盘重点

1. 这次交易是否遵守计划？
2. 盈利是否可复制？
3. 亏损是否来自策略错误？
4. 亏损是否来自执行错误？
5. 是否有 FOMO、报复性交易、无止损、加仓扛单等问题？
6. 下次应该如何调整？

## 八、实体关系概要

### 1. 自选币与交易计划

```text
WatchCoin 1 - N TradePlan
```

一个自选币可以有多个交易计划。

---

### 2. 交易计划与风险检查

```text
TradePlan 1 - N RiskCheck
```

一个交易计划可以多次执行风险检查。

---

### 3. 交易计划与仓位计算

```text
TradePlan 1 - N PositionSizing
```

一个交易计划可以多次计算仓位。

---

### 4. 交易计划与交易记录

```text
TradePlan 1 - 0..1 TradeRecord
```

一个计划可以最终生成一条真实交易记录。

但也允许某些交易记录没有计划，用于记录历史交易。

---

### 5. 交易记录与复盘

```text
TradeRecord 1 - 0..1 Review
```

一个交易记录可以有一条复盘。

---

### 6. 市场快照与 AI 分析

```text
MarketSnapshot 1 - N AIAnalysis
```

同一个市场快照可以被 GPT 多次分析。

---

### 7. 市场快照与用户判断

```text
MarketSnapshot 1 - N UserConclusion
```

一个市场快照可以关联多次用户判断。

---

### 8. AI 分析与参数推荐

```text
AIAnalysis 1 - N ParameterRecommendation
```

一次 AI 分析可以产生多个参数建议。

---

### 9. 策略与策略执行

```text
Strategy 1 - N StrategyRun
```

一个策略模板可以被执行多次。

---

### 10. 策略执行与因子信号

```text
StrategyRun 1 - N FactorSignal
```

一次策略执行可以包含多个因子信号。

---

### 11. 策略与因子配置

```text
Strategy N - N Factor
```

通过 StrategyFactor 连接。

---

### 12. 因子与权重历史

```text
Factor 1 - N FactorWeightHistory
```

一个因子可以有多条权重变化历史。

---

## 十、给 Codex 的实现要求

后续让 Codex 基于本文档实现 Prisma schema 时，应遵守：

1. 使用 SQLite 作为本地开发数据库。
2. 所有金额、价格、比例使用 Decimal。
3. 所有枚举使用 Prisma enum。
4. 不新增不必要依赖。
5. 不接交易所 API。
6. 不实现自动交易。
7. 不保存任何敏感密钥。
8. 所有实体都需要 createdAt。
9. 需要修改的实体增加 updatedAt。
10. 风险评分逻辑单独放在 service 中。
11. 仓位计算逻辑单独放在 service 中。
12. GPT 分析模块先只保存结果，不急着接真实 API。
13. 因子权重调整第一版只生成建议，不自动调整。

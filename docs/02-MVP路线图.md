# MVP 路线图

## 一、文档目的

本文档用于定义 DigitalCoin 第一阶段 MVP 的开发范围、模块优先级、开发顺序和验收标准。

DigitalCoin 的长期目标是构建一个加密货币交易辅助、AI 分析、策略生成、风控复盘与因子权重演化系统。

但是 MVP 阶段不追求一次性完成所有能力，而是先完成最小闭环：

```text
自选币
  ↓
交易计划
  ↓
风险检查
  ↓
仓位计算
  ↓
用户手动执行
  ↓
交易记录
  ↓
复盘总结
```

第一版先让系统“能用起来”，再逐步加入 AI 分析、行情数据、策略因子和权重演化。

---

## 二、MVP 开发原则

MVP 阶段必须遵守以下原则：

1. 不接交易所私有 API
2. 不自动下单
3. 不自动转账
4. 不自动提现
5. 不保存钱包私钥
6. 不保存助记词
7. 不做实盘交易机器人
8. 不做高频交易
9. 不追求复杂行情系统
10. 优先完成交易前检查、风险评分、仓位计算、交易记录和复盘闭环

MVP 的目标不是“预测行情”，而是帮助用户减少冲动交易和错误交易。

---

## 三、MVP 总体分期

DigitalCoin 可以按以下阶段推进：

| 阶段 | 名称 | 目标 |
|---|---|---|
| Phase 0 | 项目初始化 | 完成项目基础设施、文档、Git、安全规则 |
| Phase 1 | 手动录入版 MVP | 完成核心交易辅助闭环 |
| Phase 2 | AI 分析记录版 | 接入 GPT 分析记录，但不自动采集所有数据 |
| Phase 3 | 策略与因子版 | 引入策略模板、因子信号、权重记录 |
| Phase 4 | 公共数据增强版 | 接入公共行情、宏观、合约、链上数据 |
| Phase 5 | 只读账户集成版 | 可选接入只读交易所 API |
| Phase 6 | 自动交易评估版 | 仅在安全和复盘体系成熟后评估 |

3. 风险检查

### 4. RiskCheck 风险检查模块

#### 功能目标

根据交易计划自动计算风险评分和风险等级。

#### 初始评分规则

| 风险项 | 分值 |
|---|---:|
| 没有止损 | +25 |
| 合约且杠杆大于 3 倍 | +20 |
| 合约且使用全仓 | +25 |
| 情绪状态为 FOMO、想回本、冲动 | +25 |
| 有重大宏观事件 | +15 |
| 追涨 | +15 |
| 最大亏损超过总资金 2% | +20 |

#### 风险等级

| 分数 | 风险等级 |
|---|---|
| 0 - 30 | 低风险 |
| 31 - 60 | 中风险 |
| 61 - 80 | 高风险 |
| 81 - 100 | 极高风险 |

#### 输出内容

1. 风险评分
2. 风险等级
3. 触发的风险原因
4. 系统建议
5. 是否建议继续操作

#### 极高风险提示

如果风险等级为极高风险，系统应提示：

```text
不建议操作。
该交易可能存在过度杠杆、无止损、情绪化交易或重大事件风险。
```

#### 验收标准

1. 系统可以根据 TradePlan 自动生成风险分
2. 风险原因必须清晰展示
3. 极高风险必须有明显提示
4. 风险检查结果需要保存历史记录

### 5. PositionSizing 仓位计算模块

#### 功能目标

帮助用户在交易前明确最大亏损和盈亏比。

#### 输入字段

1. 当前总资金
2. 计划投入金额
3. 杠杆倍数
4. 入场价
5. 止损价
6. 止盈价

#### 输出字段

1. 实际仓位价值
2. 止损亏损金额
3. 止盈盈利金额
4. 盈亏比
5. 止损亏损占总资金比例
6. 仓位建议

#### 计算说明

实际仓位价值：

```text
实际仓位价值 = 计划投入金额 × 杠杆倍数
```

多单止损亏损估算：

```text
亏损金额 = 实际仓位价值 × (入场价 - 止损价) / 入场价
```

多单止盈盈利估算：

```text
盈利金额 = 实际仓位价值 × (止盈价 - 入场价) / 入场价
```

盈亏比：

```text
盈亏比 = 预期盈利金额 / 预期亏损金额
```

止损亏损占总资金比例：

```text
止损亏损占比 = 止损亏损金额 / 总资金 × 100%
```

#### 验收标准

1. 系统可以计算止损亏损金额
2. 系统可以计算止盈盈利金额
3. 系统可以计算盈亏比
4. 如果止损亏损超过总资金 2%，系统应提示仓位过重
5. 金额计算必须使用高精度 Decimal，不能使用普通 float

## 七、Phase 2：AI 分析记录版

### 阶段目标

接入 GPT 分析能力，但先不追求全自动数据采集。

该阶段重点是建立：

```text
用户判断
  ↓
GPT 分析
  ↓
判断对比
  ↓
交易计划
  ↓
结果复盘
```

### 功能清单

1. 用户手动输入市场信息
2. 用户记录自己的判断
3. 系统调用 GPT 分析
4. 保存 GPT 分析结果
5. 对比用户判断和 GPT 判断
6. 将 GPT 建议转化为交易计划草稿
7. 交易结束后复盘 GPT 判断是否有效

### AI 分析类型

1. 行情分析
2. 网格参数分析
3. 合约风险分析
4. 山寨币机会分析
5. 复盘辅助分析

### 验收标准

1. 用户可以保存自己的判断
2. 系统可以保存 GPT 的分析结果
3. 系统可以对比用户判断和 GPT 判断
4. GPT 输出不能直接触发交易
5. GPT 生成的策略建议必须经过 RiskCheck

## 九、Phase 4：公共数据增强版

### 阶段目标

逐步接入公共数据源，提高分析效率。

### 可接入数据源

1. 交易所公共行情 API
2. TradingView 相关数据或截图分析
3. Trading Economics 宏观日历
4. CME FedWatch 利率预期
5. Coinglass 合约数据
6. DeFiLlama 稳定币和 TVL 数据
7. Arkham 链上钱包数据
8. Bitcoin Treasuries 机构 BTC 持仓
9. The Block Treasuries 企业山寨币持仓
10. RWA.xyz RWA 数据

### 接入原则

1. 优先使用公共 API
2. 优先使用不需要密钥的数据源
3. 如果需要 API Key，必须只读
4. 不接提现权限
5. 不把 Secret 暴露到前端
6. 不把真实密钥发送给 GPT

### 验收标准

1. 至少接入一个公共行情数据源
2. 数据保存为 MarketSnapshot
3. 数据可以作为 GPT 分析输入
4. 数据源和更新时间需要记录
5. 数据不可用时不能影响核心手动流程

## 十、Phase 5：只读账户集成版

### 阶段目标

在安全可控的前提下，可选接入交易所只读账户 API。

### 可读取内容

1. 总资产
2. 可用余额
3. 现货持仓
4. 合约仓位
5. 未实现盈亏
6. 历史订单
7. 手续费
8. 资金费率支出

### 安全要求

1. API Key 必须只读
2. 禁止提现权限
3. 禁止交易权限
4. API Secret 只允许保存在服务端环境变量
5. 前端不能访问 API Secret
6. 日志不能打印 API Key 或 Secret
7. 用户必须明确知道当前接入权限

### 验收标准

1. 可以读取只读资产信息
2. 不能下单
3. 不能提现
4. 不能转账
5. API Key 不暴露到前端
6. 错误日志不泄露密钥

## 十一、Phase 6：自动交易评估版

### 阶段目标

自动交易不属于当前开发范围，只作为未来评估方向。

只有满足严格条件后，才能考虑。

### 前置条件

1. 手动交易流程稳定
2. 至少完成 100 笔交易记录
3. 至少完成 100 次复盘
4. 风险评分规则稳定
5. 仓位控制规则稳定
6. 策略因子权重有足够样本
7. 有模拟盘验证
8. 有独立安全审查
9. 有人工确认机制
10. 有一键停止功能

### 当前结论

MVP 阶段不做自动交易。

### 推荐依赖

第一阶段尽量少装依赖。

可考虑：

1. prisma
2. @prisma/client
3. zod
4. react-hook-form
5. recharts

涉及交易所、钱包、签名、自动交易的依赖，必须单独评估后再引入。

---

## 十四、MVP 验收总标准

第一版 MVP 完成后，应满足：

1. 用户可以添加自选币
2. 用户可以创建交易计划
3. 系统可以自动生成风险评分
4. 系统可以计算仓位和最大亏损
5. 用户可以记录真实交易结果
6. 用户可以为交易写复盘
7. Dashboard 可以展示基础统计
8. 系统不自动下单
9. 系统不保存任何私钥或助记词
10. 系统不接带交易权限的 API
11. 系统所有关键文档齐全
12. 所有代码已提交 GitHub
13. `pnpm audit` 无高危漏洞，或已记录原因和处理计划
14. `.env.local` 不会被提交
15. `pnpm-lock.yaml` 已提交

---

## 十五、MVP 后的下一步

当手动录入版 MVP 可用后，再进入 AI 分析和策略迭代。

后续重点不是堆功能，而是逐步增强：

1. 数据来源
2. AI 分析能力
3. 策略模板
4. 因子权重
5. 复盘质量
6. 统计分析
7. 风控精度

最终形成一个真正属于用户自己的加密货币交易辅助系统。
